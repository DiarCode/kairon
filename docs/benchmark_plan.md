# Benchmark Plan — Kairon

**Date:** 2026-06-05
**Goal:** Every shipped model beats these baselines on cost-aware Sharpe **and** is no worse on calibration. If a model can't beat LR, it doesn't ship.

## 1. Reference baselines

| Baseline | Why |
|----------|-----|
| **Persistence (close[t] → close[t+H])** | Strongest possible baseline in a near-efficient market. |
| **Linear / Logistic Regression with TA features** | Industry workhorse. |
| **XGBoost with TA features only** | Strong tabular baseline. |
| **Naive cross-asset average** | A "no model" reference. |
| **Buy & hold (per asset)** | Sanity reference. |

## 4. Headline KPIs

For a model to be "strong" (per the milestone ladder):
- DSR ≥ 0.95
- PBO ≤ 0.10
- ECE ≤ 0.05
- Cost-aware Sharpe ≥ 1.0
- Max drawdown ≤ 15%
- Hit rate × avg-win / (avg-loss × commission mult) > 1.0
- Regime breakdown non-degrading

For a model to be "breakthrough":
- Cost-aware Sharpe ≥ 1.5 sustained 12 months
- Drawdown < 12%
- DSR ≥ 0.97
- PBO ≤ 0.08
- All of the above + paper trade validation

## 5. Benchmark datasets (CI fixtures)

Stored in `evals/datasets/`:
- BTC-USDT 5-min, 2022-2023, 1 symbol.
- BTC-USDT 1h, 2018-2024.
- AAPL 1d, 2010-2024 (from TWRR).
- SPY 1d, 1993-2024 (from TWRR).
- Multi-coin crypto (10 coins) 5-min, 2022-2023 (R-002 reproduction).

Each fixture is versioned (hash) and pinned.

## 6. Benchmark runs

- `evals/runner.py` runs each (model, asset, horizon, fixture) combo.
- Output: a JSON diff vs the baseline file in `evals/baselines/`.
- CI fails on regression > tolerance.
- `evals/runner.py` is also used to run **adversarial tests**: noise-only fixtures (where no model should beat LR by much), shuffled-label fixtures (where every model should be at chance), cost-shock fixtures (where pre-cost > post-cost by a known factor).

## 7. Reporting

For every benchmark run:
- Quantitative: DSR, PBO, ECE, CAS, drawdown, regime breakdown, ablation table.
- Qualitative: a one-paragraph "what changed and why" auto-generated by the LLM (read-only, evidence-grounded).
- Both are written to `reports/{model}/{version}/`.

## 8. Live paper-trading evaluation

| Phase | Duration | Gates |
|-------|----------|-------|
| **Dry run** (signal-only, no log) | 1 week | data feeds stable, latency OK |
| **Paper trade** (logged, no real money) | 4 weeks | CAS within 1.5 std of backtest; ECE drift < 0.03 |
| **Canary live** (very small size) | 4 weeks | CAS within 1.0 std; no regime classifier failures |
| **Production** | ongoing | weekly review, monthly DSR/PBO report |

## 9. Honest verdict

> Kairon does not promise to beat published SOTA numbers in absolute terms — those are reported on authors' chosen fixtures and splits. Kairon promises to be **at least as good as our published baselines on our fixtures**, with a known and small error bar, under cost-aware walk-forward evaluation. Any deviation from this promise is auto-flagged and the offending model is held from production.
